# Showcase Platform Integration Design

## Goal

本轮工作的目标，不是把 8 个独立 HTML 佐证页原样塞回现有站点，而是以现有网站为基础，系统性吸收这些页面中最有价值的“运行、应用、成效、视频推广”信息，使整站更像一个真实在运行的平台。

这次设计要同时完成四件事：

1. 将首页从“平台门厅型首页”升级为“运营总览首页”。
2. 将成效展示页从“成果分类陈列”升级为“平台应用成效总览页”。
3. 保留顶层平台功能逻辑，而不是改成纯申报材料目录。
4. 让全站各页都服务于同一叙事：平台不是静态成果，而是正在稳定运行、被持续使用、并已形成推广影响的教学平台。

## Core Direction

### Primary site posture

整站优先服务于：

- 真实平台观感
- 实际浏览体验
- 专家审阅时的可信度与完成度

而不是：

- 截图拼贴感
- 申报材料目录感
- 说明文式成果门户

### Primary narrative

全站最重要的第一认知不是“平台有哪些功能”，而是：

- 平台已经形成应用成效与推广影响

功能结构仍然保留，但功能要作为“真实平台入口”出现，而不是首页第一叙事。

## Scope

### This round includes

- `/app/page.js`
- `/app/impact/page.js`
- `/app/cases/page.js`
- `/app/cases/[id]/page.js`
- 研习工作台相关页面与组件
- `/app/resources/page.js`
- `/app/courses/page.js`
- 顶层导航、站点路由命名与跨页信息组织
- `lib/showcase-content.js` 及其相关内容组织逻辑
- 必要的首页/成效页/视频页/平台页组件与样式重构

### This round does not include

- 新增 8 个独立静态 HTML 路由
- 把独立 HTML 页面当作 iframe、截图或附件直接嵌入站点
- 偏离现有站点骨架去重做一个完全不同的产品

## Design Decision

### Chosen approach

采用“平台优先型重构”。

即：

- 保留核心顶层平台入口
- 重排导航与页面职责
- 将 8 个 HTML 页面拆成若干个信息单元
- 重新分配到首页、成效展示页、视频页及其他相关页面中
- 必要时新增板块，但不为“补内容”而机械增块

### Rejected approaches

#### 1. 保守型整合

只对首页和成效页做轻改，其他页面不动。

不采用的原因：

- 会导致全站叙事断裂
- 8 个 HTML 的价值无法充分吸收
- 首页重构后的方向无法在其他页面继续成立

#### 2. 后台化激进重构

把整站都强推成纯运营后台风格。

不采用的原因：

- 会弱化真实教学平台的使用体验
- 案例检索与研习工作台容易失去自然的教育产品气质
- 专家看到的会更像“演示后台”，而不是“在用平台”

## Information Architecture

### Top-level navigation

重构后保留功能逻辑型顶层导航，并允许重排与改名。推荐结构如下：

1. 首页
2. 案例检索
3. 研习工作台
4. 成效展示
5. 课程视频
6. 课程体系

其中：

- `首页` 负责运营总览，不负责完整解释平台功能。
- `案例检索` 是首页后的第一优先功能入口。
- `研习工作台` 作为深度使用证明独立存在。
- `成效展示` 是独立顶层入口，不弱化为首页附属区块。
- `课程视频` 保留独立顶层入口，用于承接展示、播放与推广共享。
- `课程体系` 继续存在，但权重低于真实平台入口。

### Route posture

当前站点已有路由：

- `/`
- `/cases`
- `/cases/[id]`
- `/impact`
- `/courses`
- `/resources`

建议在不破坏现有技术骨架的情况下完成语义重组：

- `/resources` 向“课程视频 / 视频共享”方向转型
- `/courses` 保留课程体系职责
- `/impact` 升级为平台成效总览页

## Page Roles

### 1. Homepage

#### New role

首页改为“运营总览首页”。

首屏主叙事：

- 平台已经形成应用成效与推广影响

首屏之后立即进入：

1. 关键成效指标
2. 运行趋势
3. 应用覆盖

而不是先解释平台价值或功能目录。

#### What homepage should stop doing

- 不再把“案例检索 / 研习链路说明”放在首页前段
- 不再用大量平台功能导览文本占据中上部
- 不再以课程、资源、视频、成效等均质 section 平铺

#### What homepage should start doing

- 用 4 到 6 个高优先级指标卡片快速建立运行感
- 用更强的运营语言而非入口说明语言组织首页中段
- 将“真实平台入口”后置为能力支撑，而不是主叙事
- 把专家最需要先看到的信息前置：覆盖规模、使用强度、趋势与影响

### 2. Impact Page

#### New role

成效展示页升级为“独立的项目成效总览页”。

整页主轴：

- 平台应用成效

课程与视频：

- 作为支撑模块向后展开

优先模块顺序：

1. 运行趋势
2. 应用覆盖
3. 视频推广共享
4. 课程与资源支撑

#### New emphasis

成效页优先回答：

- 平台是不是持续运行
- 平台是不是被真实使用
- 平台的覆盖和使用深度如何
- 平台是否已经形成推广与共享价值

而不是先按“教学建设 / 学生发展 / 平台运行 / 推广示范”做平行成果归类。

### 3. Cases Page

#### Role after restructure

案例检索页继续作为首页后的第一功能入口。

它承担的不是“展示有案例”，而是：

- 证明平台有真实内容可检索
- 让首页后置的功能入口仍然足够强
- 与首页形成“总览 → 使用入口”的自然递进

#### Adjustment direction

- 强化“真实平台工作区”感
- 保留高可用检索入口
- 文案从“介绍案例链路”改为“帮助立即进入检索与浏览”
- 与首页运营总览形成节奏差：首页讲运行，案例页讲使用

### 4. Case Detail Page

#### Role after restructure

案例详情页继续承担导读与进入研习的职责，但需要更明确地服务于整站叙事：

- 它是平台真实使用链路中的中间层
- 不是单独的资料详情页

#### Adjustment direction

- 保持导读判断页定位
- 在视觉与文案上更明确与平台运行中的真实内容库相连接
- 弱化说明式语气，增强“继续使用平台”的自然动作

### 5. Study Workspace

#### Role after restructure

研习工作台继续保留独立功能入口，并承担：

- 平台深度使用场景证明
- 从检索到研习的真实工作流闭环

#### Adjustment direction

- 不需要前置到首页主叙事
- 但必须在全站中维持足够强的“真实工作流”可信度
- 继续突出事实梳理、争议焦点、法理分析三步结构

### 6. Video Page

#### Role after restructure

视频页必须从“资源页的一部分”升级为独立模块。

它承担：

- 示范性教学视频展示
- 播放与共享入口
- 推广传播证据

视频页的角色不是补充，而是平台对外传播与课程延展的一条独立能力线。

### 7. Courses Page

#### Role after restructure

课程体系页继续存在，但不再是首页中段的核心展开对象。

它的职责：

- 补充平台背后的课程体系可信度
- 作为支撑页面服务于整站完成度

课程页的权重仍然低于：

- 案例检索
- 研习工作台
- 成效展示
- 课程视频

## Content Integration Strategy

### Principle

8 个 HTML 页面不是作为视觉模板复用，而是作为内容来源拆解。

吸收时遵循：

- 优先吸收“后台数据、应用成效、视频推广共享”
- 少量吸收“平台功能模块/首页栏目”作为辅助证明
- 转写成网站原生组件和信息块
- 不保留强截图感和外来页面感

### Source priority from the 8 HTML pages

主来源：

- `fig1-46-admin-overview.html`
- `fig2-22-admin-modules.html`
- `fig2-41-operations-trends.html`
- `fig3-2-usage-impact.html`
- `video-promo-sharing.html`

次来源：

- `fig2-20-home-columns.html`
- `fig2-21-modules-workspace.html`
- `fig3-1-application-page.html`

### Mapping of HTML content into the site

#### Homepage

首页优先吸收：

- 应用覆盖规模
- 累计访问等高层指标
- 关键行为质量指标
- 运行趋势中的最能建立“持续运行感”的摘要
- 推广影响中的最能建立“形成外溢价值”的摘要

首页只少量保留：

- 平台功能模块概览
- 平台使用路径的缩略说明

#### Impact page

成效页优先吸收：

- 趋势图与趋势叙事
- 应用覆盖与行为指标组合
- 高活跃模块说明
- 活跃用户相关指标
- 视频推广与共享数据
- 平台运行说明与内容承载能力

#### Cases page and study-related pages

从独立 HTML 中只吸收少量支撑信息：

- 平台模块存在性
- 平台已形成工作区与深度使用界面的证据

目的不是复制后台数据，而是增强：

- 平台功能成立度
- 页面间叙事连续性

#### Video page

重点吸收：

- 主视频位与播放列表逻辑
- 分享、传播与专题视频组织方式
- “展示 + 推广 + 共享”三重角色

## Homepage Design Framework

### Homepage sequence

推荐首页结构如下：

1. Hero / 总览首屏
2. 关键成效指标
3. 运行趋势摘要
4. 应用覆盖与使用强度
5. 推广影响 / 视频共享摘要
6. 平台能力入口（案例检索 / 研习工作台 / 视频 / 成效）
7. 课程体系与资源支撑简述

### Homepage hero

首屏不再用“平台门厅 + 功能面板”的组合，而是更接近：

- 平台总览标题
- 一段克制说明
- 成效与影响导向的数字摘要
- 一个清楚的次层操作区或状态区

### Homepage metrics block

指标模块采用“4 到 6 个大数字卡片”为主。

指标构成分为两层：

#### Layer 1: 覆盖与访问

- 累计访问
- 平台内容总量或专题数量

#### Layer 2: 行为质量

- 案例检索使用占比
- 工作台回访率
- 视频观看或分享表现

### Homepage ability section

首页仍应保留平台能力入口，但位置后置。

这里不是首页第一叙事，而是用于说明：

- 成效不是抽象结果，而是建立在真实平台能力之上的

## Impact Page Framework

### Page sequence

推荐成效页结构如下：

1. 成效页总览首屏
2. 运行趋势主模块
3. 应用覆盖主模块
4. 使用行为与高活跃模块
5. 视频推广共享
6. 课程与资源支撑
7. 收束性总结

### Trend module

这是成效页的第一核心展开模块。

应重点建立：

- 平台持续运行
- 内容持续更新
- 活跃变化可见

### Coverage module

这是成效页的第二核心模块。

强调：

- 平台被多少人使用
- 访问到什么程度
- 使用强度如何

### Video support module

视频模块在成效页中不做第一主角，但必须出现，因为它能补足：

- 推广共享
- 课程传播
- 成果外溢能力

## Cross-site Content Rules

### Tone

文案气质统一为：

- 正式
- 克制
- 平台化
- 面向真实访客和专家审阅者

避免：

- 截图说明口吻
- “本图展示”“本页用于”等申报式旁白
- 设计解释语气
- 明显为了评审而写的防御性文字

### Visual translation

独立 HTML 中的内容需要翻译为站内原生表达：

- 原生卡片
- 原生指标区
- 原生趋势区
- 原生视频模块
- 原生导航与页面段落

不采用：

- 截图拼贴
- 模拟浏览器壳
- 强行保留后台截图结构

## Execution Boundaries

### Allowed

- 明显调整顶层导航结构
- 重写首页与成效页的主结构
- 重写部分页面职责与排序
- 将 `/resources` 语义转向视频展示与共享
- 扩展内容模型以支撑新的运营总览叙事

### Not allowed

- 为了吸收 8 个 HTML 而让整站变成材料拼贴页
- 为了证明“后台运行”而把所有页面都硬做成数据后台
- 让首页失去平台入口价值
- 把案例检索与研习工作台合并成一个模糊入口

## Verification

实现后至少要完成以下验证：

1. 首页第一屏能在数秒内建立“平台已经形成应用成效与推广影响”的认知。
2. 首页首屏后的第一主模块确实是关键成效指标，而不是平台功能介绍。
3. 成效页第一核心模块确实是运行趋势，而不是泛化成果分类。
4. 全站导航仍然是功能逻辑型导航，而不是成果目录型导航。
5. 案例检索、研习工作台、课程视频、成效展示四个入口角色清楚，不彼此抢位。
6. 独立 HTML 中的核心数据与叙事已经被吸收到站内，但站点仍像原生平台，而不是 HTML 拼装体。
7. 首页与成效页在桌面端截图时具备强说服力，同时在实际浏览时也成立。
